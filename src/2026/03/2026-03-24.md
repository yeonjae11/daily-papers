## 📝 상세 분석

### 1. Orla: A Library for Serving LLM-Based Multi-Agent Systems

**트랙**: ML Systems | **링크**: [arXiv](https://arxiv.org/abs/2603.13605)

📋 Problem Definition

현재 LLM 기반 에이전틱 시스템은 여러 LLM 추론 단계, 도구 호출, 이종 인프라를 결합한 워크플로우로 구성되지만, 기존 서빙 시스템(vLLM, SGLang 등)은 개별 요청 수준의 최적화만 제공한다. 워크플로우 내 스테이지 간 이질성(서로 다른 모델/백엔드 요구사항)과 스테이지 간 조율(스케줄링, 컨텍스트 전파, KV 캐시 수명 관리) 문제를 해결할 메커니즘이 없어, 개발자가 오케스트레이션 코드를 수동으로 구성해야 하는 상황이다. Orla는 요청 수준 실행과 워크플로우 수준 정책을 분리하여 이 간극을 메우는 서빙 라이브러리를 제안한다.

📚 Background / Related Works

LLM 서빙 엔진으로는 SGLang(RadixAttention 기반 KV 캐시 관리)과 vLLM(PagedAttention, 연속 배칭)이 있으나 이들은 개별 요청 수준 최적화에 집중한다. 에이전트 오케스트레이션 프레임워크인 LangGraph와 AutoGen은 멀티에이전트 워크플로우 구성을 지원하지만 서빙 레이어를 블랙박스로 취급하여 동적 라우팅이나 KV 캐시 수명 관리 같은 서빙 수준 결정을 내리지 못한다. 가장 유사한 연구인 ThunderAgent(Kang et al., 2026)는 에이전틱 워크플로우를 상태기반 프로그램으로 추상화하고 KV 캐시 스래싱을 완화하지만, 단일 모델의 평탄한 시퀀스만 다루며 Orla처럼 DAG 기반 멀티모델/멀티백엔드 라우팅은 지원하지 않는다.

🔬 Main Methodology

Orla의 핵심 설계는 세 가지 컴포넌트로 구성된다. 첫째, Stage Mapper는 각 워크플로우 스테이지를 적절한 모델과 백엔드에 매핑하는데, 개발자가 명시적으로 지정하거나 OneBitStageMapper처럼 경량 LLM 호출로 요청을 "simple/complex"로 분류하여 동적 라우팅하는 방식을 지원한다. 둘째, Workflow Orchestrator는 2단계 스케줄링(스테이지 큐 선택 후 큐 내 요청 선택)과 컨텍스트 전파를 담당하며, DAG 의존성을 준수하면서 준비된 스테이지를 병렬 디스패치하고 빌더 함수를 통해 상위 스테이지 출력을 하위 스테이지 프롬프트로 전달한다. 셋째, Memory Manager는 개별 요청이 아닌 워크플로우 수준에서 KV 캐시 수명을 관리하여, 같은 백엔드에서 연속 스테이지가 컨텍스트를 공유할 때 캐시를 보존하고, 백엔드 전환이나 워크플로우 완료 시 캐시를 해제하며, 메모리 압박 시 유휴 워크플로우의 캐시를 축출하는 세 가지 정책을 구현한다. 워크플로우는 DAG로 표현되며, 모든 백엔드는 OpenAI 호환 HTTP API를 통해 플러그앤플레이 방식으로 접근되어 SGLang, vLLM, Ollama, 시뮬레이터 간 전환이 엔드포인트 URL 변경만으로 가능하다.

🧪 Evaluation

SWE-bench Lite(2,294개 요청)에서 OneBitStageMapper를 사용해 956개를 Qwen3-4B로, 나머지를 Qwen3-8B로 라우팅한 결과, 단일 모델(Qwen3-8B on vLLM) 대비 전체 실행 시간 38% 감소, 평균 완료 시간 60% 감소, 추정 추론 비용 35% 절감을 달성했다. DAG-MATH-Formatted-CoT(2,894개 요청) 데이터셋에서 SGLang 백엔드의 Qwen3-8B를 사용해 캐시 관리 실험을 수행한 결과, 요청별 캐시 플러시 대비 워크플로우별 캐시 플러시 정책이 전체 TTFT(Time-to-First-Token) 분포를 개선하며 KV 캐시 재사용을 통해 불필요한 프리필을 회피했다. 실험 환경은 단일 머신(AMD EPYC 7313 x2, 528GB RAM, NVIDIA RTX PRO 6000 Blackwell 96GB VRAM)이다.

💡 Key Intuition & Lesson

핵심 인사이트는 에이전틱 워크플로우의 최적화 단위가 개별 LLM 요청이 아닌 워크플로우 전체여야 한다는 것이다. 개별 요청 수준에서는 보이지 않는 신호들(스테이지 의존성, 예측 재사용 패턴, 백엔드 전환, 워크플로우 완료)이 KV 캐시 관리와 스케줄링에 결정적이며, 이를 활용하면 단순히 "큰 모델 하나로 모든 것을 처리"하는 방식 대비 지연시간과 비용을 동시에 크게 줄일 수 있다. 또한 요청 실행과 워크플로우 정책을 깔끔하게 분리하는 추상화 계층을 두면, 기존 서빙 엔진이나 오케스트레이션 프레임워크를 수정하지 않고도 그 사이에서 상당한 최적화 기회를 포착할 수 있다는 실용적 교훈을 제공한다.
