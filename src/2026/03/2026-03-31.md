## 📝 상세 분석

### 1. Understanding the Challenges in Iterative Generative Optimization with LLMs

**트랙**: Agents | **링크**: [arXiv](https://arxiv.org/abs/2603.23994)

제공된 abstract와 메타데이터를 기반으로 분석하겠습니다.

📋 Problem Definition

LLM을 활용한 생성적 최적화(Generative Optimization)는 코드, 워크플로우, 프롬프트 등의 artifact를 실행 피드백을 통해 반복적으로 개선하는 접근법이지만, 실제로는 매우 취약(brittle)하여 조사 대상 에이전트 중 9%만이 자동화된 최적화를 사용하고 있다. 이 논문은 이러한 취약성의 근본 원인이 학습 루프를 설정할 때 엔지니어가 암묵적으로 내려야 하는 "숨겨진 설계 결정(hidden design choices)"에 있다고 주장하며, 이를 체계적으로 분석하고 실용적 가이드라인을 제시하고자 한다.

📚 Background / Related Works

이 연구는 자기 개선 에이전트(self-improving agents) 패러다임에 속하며, LLM 기반 코드 생성 벤치마크인 MLAgentBench, 강화학습 환경인 Atari, 그리고 추론 벤치마크인 BigBench Extra Hard(BBEH) 등 기존 평가 프레임워크를 활용한다. 생성적 최적화는 FunSearch, EvoPrompting 등 LLM을 진화적/반복적 최적화기로 활용하는 선행 연구들과 맥을 같이하며, 프롬프트 최적화(prompt optimization), LLM 기반 코드 에이전트(code agents), 그리고 실행 피드백 기반 자동 개선(automated refinement with execution feedback) 분야의 활발한 연구들이 배경이 된다. 특히 이 논문은 기존 연구들이 이러한 설계 결정을 명시적으로 다루지 않았다는 점을 문제로 지적한다.

🔬 Main Methodology

이 논문은 생성적 최적화 루프의 성패를 결정짓는 세 가지 핵심 설계 요인을 식별하고 체계적으로 실험한다. 첫째, 시작 artifact(starting artifact)는 최적화기가 편집할 수 있는 대상과 그 초기 상태로, 어떤 시작점에서 출발하느냐에 따라 도달 가능한 해(solution)의 공간이 근본적으로 달라진다. 둘째, 크레딧 호라이즌(credit horizon)은 실행 트레이스에서 얼마만큼의 범위를 학습 신호로 제공할 것인지에 대한 결정으로, 전체 트레이스를 줄 것인지 혹은 잘라낸(truncated) 트레이스로도 충분한지를 다룬다. 셋째, 미니배치(minibatch) 구성은 시행착오(trials and errors)를 학습 증거로 묶는 방식으로, 배치 크기가 일반화 성능에 미치는 영향을 탐구한다. 이 세 요인을 MLAgentBench(코드 생성), Atari(게임 에이전트), BBEH(추론 태스크)라는 서로 다른 도메인의 사례 연구를 통해 각각 독립적으로 검증하는 구조를 취한다.

🧪 Evaluation

MLAgentBench에서는 서로 다른 시작 artifact가 최적화를 통해 도달할 수 있는 해의 범위를 결정적으로 변화시키는 것을 확인했다. Atari 환경에서는 잘라낸(truncated) 실행 트레이스만으로도 에이전트 성능을 개선할 수 있음을 보여주어, 전체 트레이스가 반드시 필요하지 않다는 점을 실증했다. BBEH에서는 미니배치 크기를 키우는 것이 일반화 성능을 단조적으로(monotonically) 향상시키지 않으며, 오히려 특정 지점 이후에는 성능이 저하되거나 정체되는 현상을 관찰했다.

💡 Key Intuition & Lesson

생성적 최적화는 단순히 LLM과 피드백 루프를 연결하면 작동하는 것이 아니라, 시작 artifact의 선택, 실행 트레이스의 범위 설정, 학습 증거의 배치 구성이라는 세 가지 숨겨진 설계 결정이 성패를 좌우하며, 이 결정들이 도메인마다 다르게 작용하기 때문에 범용적이고 단순한 학습 루프 설정 방법이 존재하지 않는다는 것이 핵심 교훈이다. 이는 생성적 최적화의 산업적 채택(productionization)을 위해서는 이러한 설계 결정들을 명시적으로 문서화하고 도메인별로 체계적인 가이드라인을 마련해야 한다는 실질적 시사점을 제공한다.

### 2. FinMCP-Bench: Benchmarking LLM Agents for Real-World Financial Tool Use under the Model Context Protocol

**트랙**: Agents | **링크**: [arXiv](https://arxiv.org/abs/2603.24943)

📋 Problem Definition

LLM 에이전트가 금융 도메인에서 외부 도구를 호출하며 다단계 추론을 수행하는 능력을 체계적으로 평가할 벤치마크가 부재했다. 기존 금융 LLM 평가는 특정 태스크에 한정되어 있고 도구 사용(tool use)을 포함하지 않아, 실제 금융 현장에서 요구되는 복잡한 도구 체이닝과 의존성 처리 능력을 측정할 수 없었다. FinMCP-Bench는 Anthropic이 제안한 Model Context Protocol(MCP) 기반의 65개 실제 금융 도구를 활용하여, 현실적이고 난이도 높은 금융 시나리오에서 LLM의 도구 호출 능력을 평가하기 위해 설계되었다.

📚 Background / Related Works

금융 LLM 평가 관련으로 Lei et al. (2024), Zhu et al. (2024), Nie et al. (2025), Tang et al. (2025), Xie et al. (2025) 등의 선행 연구가 있으나, 이들은 대부분 특정 금융 태스크(예: 감성 분석, 질의응답)에 초점을 맞추고 도구 호출을 포함하지 않는다. 일반 도구 사용 평가 벤치마크로는 Dong et al. (2025), Li et al. (2025), Qian et al. (2025), Goldie et al. (2025) 등이 있으나 이들은 최종 답변 정확도에 집중하며 금융 도메인 특화가 아니다. 본 논문은 Anthropic (2024)의 MCP 표준을 채택하여, 금융 도메인에서 도구 호출 정확도 자체를 평가 대상으로 삼는 최초의 벤치마크를 제안한다.

🔬 Main Methodology

FinMCP-Bench는 10개 주요 시나리오(시장 분석, 투자 계획, 리스크 관리 등)와 33개 하위 시나리오에 걸쳐 613개 샘플을 포함하며, 단일 도구(145개), 다중 도구(249개), 다중 턴(219개) 세 유형으로 구성된다. 데이터는 Yingmi Fund의 Qieman APP에서 수집한 10,000건의 실제 금융 AI 어시스턴트 로그를 기반으로 하며, 엄격한 익명화와 품질 필터링을 거쳤다. 다중 도구 샘플은 "체인 기반 합성" 방식으로 구축하는데, 먼저 65개 도구 노드와 288개 의존성 간선으로 구성된 도구 의존성 그래프를 생성하고, 그래프에서 도구 체인을 샘플링한 뒤 Qwen3-235B를 사용해 해당 체인에 부합하는 사용자 질의와 실행 궤적(trajectory)을 생성한다. 다중 턴 샘플은 역할극(role-playing) 방식으로, 페르소나 풀에서 사용자 프로필을 샘플링하고 Qwen3-235B가 사용자와 어시스턴트 역할을 동시에 수행하여 대화를 시뮬레이션한다. 최종 품질 보증은 자동 검증 후 6명의 금융 도메인 전문가가 5점 Likert 척도로 5개 차원(질문 관련성, 도구 체인 완전성, 논리적 일관성, 답변 신뢰성, 데이터 신선도)을 평가하여, 두 리뷰어 모두 4점 이상인 샘플만 채택하는 이중 검증 파이프라인을 적용했다.

🧪 Evaluation

6개 LLM(Qwen3-4B/30B/235B-Thinking, DeepSeek-R1, GPT-OSS-20B, Seed-OSS-36B)을 대상으로 Tool Precision(TP), Tool Recall(TR), Tool F1(TF1), Exact Match Rate(EMR) 네 가지 메트릭으로 평가했다. 전체 성능에서 Qwen3-235B-A22B-Thinking이 TF1 64.27%, EMR 25.92%로 가장 높았으며, 특히 다중 도구 샘플에서 TF1 69.42%를 달성했다. 다중 턴 샘플에서는 모든 모델이 EMR 5% 미만으로 급락했는데(Qwen3-30B가 4.10%, Qwen3-235B가 3.08%), 이는 긴 대화에서 정확한 도구 조합과 순서를 유지하는 것이 현재 모델들에게 극도로 어려운 과제임을 보여준다. 흥미롭게도 난이도별 분석에서 강한 모델들은 Easy보다 Hard에서 TF1이 오히려 상승했는데, 이는 쉬운 문제에서 불필요한 도구를 과다 호출(over-calling)하여 정밀도가 떨어지는 반면 어려운 문제에서는 풍부한 제약 조건이 도구 선택을 가이드해주기 때문이다.

💡 Key Intuition & Lesson

이 논문의 핵심 인사이트는 금융 에이전트 평가에서 "최종 답변의 정확도"가 아닌 "도구 호출 자체의 정확도"를 측정해야 한다는 점이다. 금융 리서치나 자문 시나리오에서는 정답이 열린 형태(open-ended)이므로, 올바른 도구를 올바른 순서로 호출했는지가 에이전트 능력의 더 본질적인 지표가 된다. 또한 모델 크기가 성능과 단순 비례하지 않으며(Qwen3-4B가 EMR에서 30B를 능가), 단일 도구 호출에서조차 모델들이 불필요한 도구를 과다 호출하는 경향이 있다는 발견은, 도구 선택의 "억제력(restraint)"이 도구 사용 능력만큼이나 중요한 역량임을 시사한다.

### 3. Hydraulis: Balancing Large Transformer Model Training via Co-designing Parallel Strategies and Data Assignment

**트랙**: ML Systems | **링크**: [arXiv](https://arxiv.org/abs/2412.07894)

📋 Problem Definition

대규모 Transformer 모델 학습 시, 실제 데이터셋의 시퀀스 길이가 매우 불균일하여 두 가지 불균형 문제가 발생한다. 첫째, data sampling imbalance로서 미니배치 내/간에 시퀀스 길이 분포가 크게 변동하여 고정된 병렬 전략이 비효율적이 되고, 둘째, data packing imbalance로서 attention 메커니즘의 시간 복잡도(이차)와 메모리 복잡도(선형)의 불일치로 인해 패킹된 시퀀스 간 실행 시간이 불균형해진다. Hydraulis는 병렬 전략과 데이터 할당을 공동 최적화하여 이 두 불균형을 동시에 해결한다.

📚 Background / Related Works

기존 대규모 모델 학습 시스템인 Megatron-LM(Shoeybi et al., 2019)과 DeepSpeed(Rasley et al., 2020)는 DP, TP, PP, CP 등 다양한 병렬 기법과 ZeRO 기반 모델 상태 샤딩을 지원하지만, 학습 전체에 걸쳐 하나의 정적이고 동질적인 병렬 전략만 사용할 수 있다는 한계가 있다. HotSPa(Ge et al., 2024)는 반복 내에서 길이별로 다른 사전 정의 병렬 전략을 전환할 수 있으나, 이질적 전략을 지원하지 못하고 전환 오버헤드가 크다. 데이터 패킹 측면에서는 Krell et al.(2022), NVIDIA NeMo(2023) 등이 max-length packing을 사용하여 패딩 낭비를 줄이지만, FlashAttention의 varlen 인터페이스에서 발생하는 연산량 불균형(sum(Si^2) 기반 이차 복잡도)을 고려하지 않아 파이프라인 간 워크로드 불균형이 남는다.

🔬 Main Methodology

Hydraulis의 핵심은 다섯 가지 기여로 구성된다. 첫째, dynamic heterogeneous parallel strategies를 도입하여 각 반복마다 시퀀스 길이 분포에 맞는 이질적 병렬 전략을 적용한다. 즉 짧은 시퀀스는 빠르지만 메모리 소모가 큰 병렬 스킴으로, 긴 시퀀스는 느리지만 메모리 효율적인 스킴으로 처리하여 GPU 활용도를 극대화한다. 둘째, optimization-propagation disaggregation 아키텍처를 제안하여 최적화 단계(그래디언트 동기화, 모델 업데이트)와 전파 단계(순전파/역전파)를 분리한다. 최적화 단계에서는 일관된 모델 상태 샤딩을 유지하면서, 전파 단계에서는 임의의 이질적 병렬 전략으로 전환할 수 있도록 LNK(link) 연산자를 설계하여 파라미터 풀링과 그래디언트 푸싱을 커스텀 통신으로 수행한다. 셋째, subgraph 추상화를 도입하여 계산 그래프의 부분집합으로 전파 중 활성화 통신과 최적화-전파 간 파라미터/그래디언트 교환을 통합 표현하며, 이를 통해 병렬 스킴의 유연한 조합과 전략 간 원활한 전환을 지원한다. 넷째, two-stage sequence assignment를 통해 패킹 불균형을 해소하는데, 1단계에서 각 파이프라인 내 마이크로배치 간 워크로드 균형을 맞추는 최적화 문제를 풀고, 2단계에서 원본 시퀀스를 파이프라인 간에 균형 있게 분배한다. 다섯째, 전체 데이터셋의 시퀀스 길이 분포를 고려한 동적 프로그래밍 기반 후보 전략 사전 생성(data distribution-aware strategy proposal)으로 반복별 전략 선택을 가속한다.

🧪 Evaluation

Hydraulis는 Hetu 프레임워크 위에 구현되었으며, LLaMA 13B 모델을 CommonCrawl 및 GitHub 데이터셋으로 Nvidia A800 GPU 16대에서 32K 컨텍스트 길이, 반복당 100K 토큰 설정으로 학습하여 평가하였다. 실험 결과 기존 최신 시스템(Megatron-LM, DeepSpeed, HotSPa 등) 대비 1.32배에서 2.66배의 학습 처리량 향상을 달성하였으며, 더 나은 확장성도 보여주었다. 또한 길이 기반 정렬 샘플링이 랜덤 샘플링 대비 perplexity에서 2배 격차가 발생하는 수렴 저하를 보인 반면, Hydraulis는 랜덤 샘플링을 유지하면서 모든 그래디언트를 동기화하여 수렴 품질에 유의미한 편차가 없음을 확인하였다.

💡 Key Intuition & Lesson

이 논문의 핵심 통찰은 대규모 모델 학습에서 병렬 전략 설계와 데이터 관리를 분리하여 다루는 기존 접근이 근본적으로 비효율적이며, 두 관점을 공동 최적화(co-design)해야 실질적인 성능 향상을 얻을 수 있다는 것이다. 특히 attention 메커니즘의 시간 복잡도(이차)와 메모리 복잡도(선형)의 비대칭성이 패킹 후 워크로드 불균형의 근본 원인이라는 분석은, 단순히 메모리를 균등하게 맞추는 것이 연산 시간 균형을 보장하지 않는다는 중요한 교훈을 제공한다. 또한 학습 수렴을 위해 랜덤 샘플링을 반드시 유지해야 한다는 실험적 증거는, 시스템 최적화가 통계적 학습 원칙을 훼손해서는 안 된다는 점을 재확인시켜 준다.
